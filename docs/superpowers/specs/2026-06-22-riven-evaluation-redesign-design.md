## Problem Statement

Текущая оценка ривенов (`backend/src/alecaframe_api/wfm/rivens_analysis.py`) — чисто
рыночная и **не знает о конкретном оружии**. `classify_tiers` режет live-аукционы
на god/mid/low по квартилям цены buyout, с ручными override'ами по захардкоженным
множествам `GOD_POSITIVES / FATAL_NEGATIVES / HARMLESS_NEGATIVES`.

Из-за этого:
- Critical Chance одинаково «god» для критического Kuva Bramma и для статусного
  Kuva Nukor — хотя для статусника CC почти бесполезен.
- Не учитываются base stats оружия (CC/CD/status/multishot/урон), disposition,
  и числовое **значение** ролла (+180% CD оценивается так же, как +90% CD).
- Не учитывается Incarnon: у многого оружия Incarnon-форма меняет архетип
  (Torid: статусный лаунчер в базе → крит-монстр в Incarnon), и на эндгейме
  играют именно через Incarnon — то есть текущая логика врёт на самых ценных
  ривенах.

Появившаяся таблица `item_base_stats` (загрузчик
`backend/src/alecaframe_api/reference/stats_loader.py`) делает редизайн возможным:
есть per-weapon CC, CD, status, multishot, разбивка урона и **disposition** по
`uniqueName`.

## What Makes This Cool

Оценка ривена становится **контекстной**: «этот ривен — S-tier для Incarnon-Torid,
но C-tier для базовой формы», с объяснением *почему*, и наложенным рыночным
слоем «и при этом он недооценён на N% относительно медианы». Один движок отвечает
сразу на два вопроса игрока: «хорош ли стат под это оружие» и «справедлива ли цена».

## Constraints

- **Стек**: FastAPI + чистые функции анализа (по образцу текущего модуля — no I/O,
  no async в анализе), SQLite history (`data/wfm_history.db`), фронт SolidJS.
- **Scope M1 = только огнестрел (primary/secondary/sentinel/arch-gun).** Melee
  ривены (range, attack_speed, combo, slide-crit, heavy, finisher) — отдельный
  stat-track, в M2. Модель данных допускает melee-архетип, но веса/словарь стат в
  M1 ствольные.
- **Источник base stats**: WFCD warframe-items (уже подключён). **Incarnon-статов
  в WFCD нет** (проверено: у 194 primary ни одного ключа `incarnon/evolution/modes`;
  Torid отдаёт только базовый профиль). → Incarnon-данные **курируются вручную**.
- **Билд игрока неизвестен** — поэтому headline-метод не может быть build-aware
  DPS-симуляцией; используем архетип-веса.
- Сохранить обратную совместимость рыночной инфры (poller, `riven_snapshot`,
  outliers) — она переосмысляется, не выбрасывается.
- Только Windows-машина с AlecaFrame; локальный инструмент.

## Premises (agreed in session)

1. Скор зависит от конкретного оружия, не от глобальных множеств стат. ✅
2. Оценивается и **наличие**, и **числовое значение** ролла относительно
   теоретического диапазона. ✅
3. Рыночная инфра сохраняется и переосмысляется как «слой справедливой цены». ✅
4. Incarnon-данные курируются вручную; архитектура хранит «набор профилей» на
   оружие с самого начала. ✅
5. Старые `GOD_POSITIVES / FATAL_NEGATIVES / HARMLESS_NEGATIVES` уходят в архив. ✅

## Approaches Considered

### Approach A — Quality core on B-architecture (Milestone 1) — CHOSEN scope
Архетип-классификатор + архетип-веса + грейд значений ролла → 0–100 + буква.
base-профили автоматом для всего оружия; Incarnon вручную для топ-торгуемых.
Рыночный слой = переосмысленная текущая инфра.
- Effort: M · Risk: Low · Reuses: весь рыночный код, snapshot history, router-shape.

### Approach B — Full profile-complete + market-calibrated (target / Milestone 2)
A + курация всех ~40 Incarnon Genesis + перк-профили + рыночная регрессия весов
(C) + «формула vs рынок» divergence-инсайт + грейд против min/max диапазонов роллов.
- Effort: L · Risk: Med (курация + шум регрессии) · Reuses: всё из A + history DB.

### Approach C — Lite-DPS scorer (alternative engine, optional)
Скор = % прироста урона на эталонном мета-билде против каждого профиля; архетип
вытекает из математики. Встаёт за тот же `score_riven` интерфейс как альтернативный
скорер/валидация.
- Effort: L–XL · Risk: Med–High (допущения о билде, краевые случаи лучи/дробовики/AoE).

## Recommended Approach

**A-scope на B-архитектуре** (= «A + B»): B — архитектура и конечная цель,
закладывается сразу; A — объём M1. C-метод (lite-DPS) откладывается за интерфейс.

## Architecture (B from day 1)

### Data model
- **`weapon_profiles`** — на каждое оружие список профилей:
  `{profile_id, weapon_slug, kind: 'base'|'incarnon'|'perk', label, effective_stats{cc,cd,status,multishot,fire_rate,damage_breakdown}, archetype, source}`.
  - `base` — генерится автоматом из `item_base_stats` для всего оружия.
  - `incarnon`/`perk` — из курируемой таблицы (новый файл данных, напр.
    `backend/src/alecaframe_api/reference/incarnon_profiles.py` или JSON в `data/`).
  - `disposition` хранится на профиле, **берётся из top-level строки** `item_base_stats`
    (не из вложенного `stats`), Incarnon-профиль наследует disposition оружия.
- **Непрерывные веса вместо жёстких архетипов (ключевое исправление).** Жёсткий
  bucket «CC≥0.20 → crit» воспроизводит ту же хрупкость, что и старые множества
  (оружие на CC 0.199 vs 0.201 «прыгает»). Вместо этого:
  - `critness = clamp(base_CC / CC_ref)`, `statusness = clamp(base_status / status_ref)`
    — непрерывные [0..1] из base stats (CC_ref/status_ref ~ верх типичного диапазона).
  - Вес архетип-чувствительных стат — **функция** от этих величин, без порогов:
    `weight(CC) ∝ critness`, `weight(CD) ∝ critness`, `weight(status) ∝ statusness`.
  - `archetype` (`crit|status|hybrid|raw_damage`) выводится из critness/statusness
    **только как ярлык для UI/объяснения**, не как переключатель логики.

### Stat vocabulary (M1, guns) — tested constant
Канонический enum статов ривена + alias-map из WFM `url_name`/`name`
(нормализация как в текущем коде: lowercase, пробелы/дефисы → `_`). Два класса:
- **Архетип-чувствительные**: `critical_chance`, `critical_damage`, `status_chance`,
  `critical_chance_on_slide` (≈crit). Вес зависит от critness/statusness.
- **Универсальные** (сильны почти на всём, вес ~архетип-независимый):
  `damage`, `multishot`, элементы `toxin/cold/heat/electricity`,
  фракция `damage_to_grineer/corpus/infested`, `fire_rate` (для не-beam).
  > `multishot` модулируется `trigger`/`slot` из base stats: для beam/AoE-лаунчеров
  > вес multishot ниже (beam масштабируется иначе). `trigger`/`slot` уже в `_norm_weapon`.
- **Утилитарные** (вес ~0 как позитив): `zoom`, `ammo_maximum`, `magazine_capacity`,
  `recoil`, `projectile_speed`, `punch_through` и т.п.

### Scoring interface (N-profile aware)
- `stat_weight(stat, profile) -> float` — для архетип-чувствительных стат функция от
  `critness/statusness` профиля; для универсальных — почти константа; масштабируется
  disposition (низкая dispo → роллы слабее → строже грейд).
- `grade_roll_value(stat, value, stat_count, has_negative, disposition) -> 0..1` —
  насколько ролл близок к теоретическому максимуму. `stat_count`/`has_negative`
  считаются из `attributes` аукциона (2 поз / 3 поз / есть негатив → разные потолки).
  - **M1 потолок**: таблица номинальных max на стат (константы, дешево, без полной
    dispo-математики) → `value` нормализуется в [0..1]. Закрывает Success #3.
  - **M2**: точные min/max диапазоны по dispo × stat_count.
- `score_riven(riven_attrs, weapon_profiles) -> RivenScore`:
  ```
  RivenScore {
    headline: { profile_id, score_0_100, grade: S|A|B|C|F },
    per_profile: [ {profile_id, kind, score, grade, breakdown[]} ],
    penalties: [ {stat, reason} ],   # фатальный негатив под архетип и т.п.
    explanation: [str]               # «CC весит много: оружие критическое (Incarnon)»
  }
  ```
  headline = профиль с максимальным баллом (по умолч. Incarnon, если он среди топа).

### Per-profile score formula (M1)
```
raw = Σ_positive( stat_weight(stat, profile) * grade_roll_value(stat, value, ...) )
    - Σ_negative( negative_penalty(stat, profile) )

# нормализация: делим на «идеальный ролл» для этого профиля —
# топ-N архетип-стат с весами этого профиля при grade_roll_value=1.0
best = ideal_roll_score(profile, n_positives=stat_count)
profile_score = clamp(100 * raw / best, 0, 100)
```
- **Грейд-кат-оффы** (фиксируем, тестируем): `S ≥ 85, A ≥ 70, B ≥ 50, C ≥ 30, F < 30`.
- **Негатив — контекстно**: `negative_penalty(stat, profile)` велик для архетип-стата
  под этот архетип (−CC на критмашине ≈ фатально), и **≈0** для off-архетип/утилитарных
  негативов (−zoom на god-ролле почти бесплатен). Заменяет статичный `FATAL/HARMLESS`;
  «мёртвый» негатив на god-ролле — это хорошо, и формула это отражает (вычитает ~0).
- **Headline-выбор**: `headline = argmax(score)`; если есть Incarnon-профиль и его
  `score ≥ max_score − ε` (ε ~5), headline = Incarnon (эндгейм-дефолт).

### Fallback (нет base-профиля)
Если оружие не нашлось в `item_base_stats` (новое/немапнутое, Zaw/Kitgun-модулярка,
провал join slug→uniqueName): `score_riven` возвращает `headline=null`,
`per_profile=[]`, флаг `unscored=true`. UI показывает только рыночный слой
+ бейдж «оценка недоступна». Покрыто тестом.

### Market layer (decoupled, pluggable weights)
- `market_signal(weapon_slug, riven_attrs) -> {fair_price_band, percentile, undervalued_pct}`.
- Источник весов рынка — интерфейс: M1 = текущие квартили/медиана/outliers из
  `riven_snapshot`; M2 = регрессия цен на наличие/значение статов из
  `wfm_history.db` (метод C).
- Комбинированный хайлайт: ривен помечается «🔥 хорош И недооценён», если
  `headline.grade ∈ {S,A}` И `undervalued_pct ≥ threshold`. Две метрики НЕ
  смешиваются в один балл — показываются рядом.

### API (router)
- `GET /rivens/auctions/{weapon_slug}` — расширяется: к каждому `RivenAuctionRow`
  добавляется `score` (RivenScore), top_attributes считается через веса, а не через
  частоту. Сохраняются tiers/stats/outliers (рыночный слой).
- (M2) `GET /rivens/score?weapon_slug=&attrs=...` — оценка произвольного ролла
  (для «мои ривены» / what-if перед рероллом).
- `GET /rivens/weapons` — добавить флаг `has_incarnon_profile`.

### Frontend (`frontend/src/routes/Rivens.tsx`)
- Карточка аукциона: буква-грейд + 0–100, тоггл профиля (base/Incarnon), раскрытие
  breakdown («почему»), бейдж «Incarnon не оценён» где курации нет.
- Хайлайт «хорош И недооценён». i18n en/ru.

## Incarnon data sourcing (key constraint)

WFCD не отдаёт Incarnon-статы. План:
- M1: курируемая таблица на топ ~10–15 Incarnon Genesis (по торгуемости: Torid, Bo,
  Latron, Dread, Boltor, Soma, Kunai, Braton, Furis, Lato, …) — поля effective_stats
  в Incarnon-режиме (CC/CD/status/multishot). Источник: Warframe Wiki, заносится руками.
  > **Упрощение M1**: Incarnon-статы зависят от выбранных evolution-перков. M1
  > профиль = «канонический max-evolution билд» (одно зафиксированное допущение,
  > указывается в label). Это сознательно вводит лёгкую build-зависимость для
  > Incarnon-профиля — приемлемо для M1; перк-варианты как отдельные профили в M2.
  > Каждое занесённое значение **валидируется** — неверный курированный CC хуже,
  > чем отсутствие профиля (уверенно-неправильный грейд).
- M2: добить все ~40 Genesis + перк-варианты. Standalone Incarnon-пушки (Laetum,
  Phenmor, Felarx, Praedos, Innodem) уже есть в WFCD как отдельные слаги — для них
  base-профиль автоматом, Incarnon-режим докуривается.
- Версионировать таблицу. Тест-страж M1 реалистичный: статический список «известных
  Incarnon-оружий» vs ключи курируемой таблицы → лог/бейдж недостающих (без обращения
  к вики в рантайме). Полный вики-aware guard — не M1.

## Success Criteria

- Для статусного оружия CC-ривен получает низкий грейд, для критического — высокий
  (регресс-тест на 2–3 эталонных оружия).
- Incarnon-оружие из курируемой таблицы скорится по Incarnon-профилю по умолчанию,
  и грейд заметно отличается от base-профиля там, где архетип меняется (Torid).
- Числовое значение ролла влияет на балл (+180% CD > +90% CD при прочих равных).
- «Мёртвый» негатив (−zoom/−recoil) на god-ролле штрафуется ≈0 и не роняет грейд;
  архетип-фатальный негатив (−CC на критмашине) роняет в F.
- Оружие без base-профиля отдаёт `unscored=true` + рыночный слой, не падает.
- Рыночный слой сохраняет текущие outliers/historical-median и добавляет
  комбинированный хайлайт «хорош И недооценён».
- Старые захардкоженные множества удалены; новый движок покрыт юнит-тестами
  (чистые функции, как и сейчас). M1 — только огнестрел; melee помечен «не оценён».
